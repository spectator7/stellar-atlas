# 天象志 v1.0 后续更新目标计划

> 基线日期：2026-07-11
> 当前阶段：`v0.6` 已完成核心星图、观测助手、故事资料库、PWA 与自动校验
> 总目标：把天象志建设成一部准确、易用、适合户外、值得反复使用，并具有鲜明中文星空特色的在线图鉴。

## 一、成功标准

`v1.0` 不以“功能数量最多”为目标，而以以下五项结果为完成标准：

1. 新用户能在一分钟内得到“今晚几点、朝哪个方向、适合看什么”的有效答案。
2. 用户在手机、弱网或离线环境中，仍能查看计划、星图和已下载资料。
3. 星图、观测计算、文化内容和来源说明具有清楚的数据口径与可复核流程。
4. 网站形成“发现目标 -> 制订计划 -> 户外观测 -> 记录结果 -> 再次探索”的使用闭环。
5. 有趣功能服务于认星、观测和文化理解，而不是以牺牲准确性、隐私和性能换取噱头。

## 二、当前基线

### 已完成能力

- 88 个现代星座及完整档案。
- 5,044 颗全天星图恒星、2,216 个深空目标、89 组边界多边形。
- 439 颗重点恒星和 211 个星座档案深空条目。
- 可缩放、拖动、筛选和键盘操作的全天 Canvas 星图。
- 46 个地点预设、手动坐标、完整 IANA 时区和可选设备定位。
- 黑暗窗口、月相月光、行星升落、流星雨、合月、天气和最佳观测时刻。
- 裸眼、双筒和小型望远镜三种器材模式及可分享观测清单。
- 12 篇专题、48 个正文分节、80 条时间线、102 个关系节点和 71 项来源。
- 本地收藏、深链接、PWA 安装、离线缓存、SEO、404 页面和 GitHub Actions 校验。

### 当前主要限制

| 用户旅程 | 当前限制 | 影响 |
| --- | --- | --- |
| 第一次访问 | 首页是一条较长的综合页面，首屏没有直接突出“今晚看什么” | 新用户需要先理解网站结构，获得结果较慢 |
| 查找目标 | 搜索主要围绕星座，不能统一搜索亮星、Messier/NGC、深空目标、专题和天象 | 数据很多，但发现效率不高 |
| 规划今晚 | 表单对新手偏复杂，没有整夜时间轴和逐目标高度曲线 | 用户知道一个推荐时刻，却不容易安排整晚顺序 |
| 户外使用 | 没有专门的红光、全屏、常亮和离线预检模式 | PWA 可以离线，但还不是成熟的户外工具 |
| 持续使用 | 清单、阅读进度和观测结果没有形成统一的本地记录 | 用户缺少回访理由和成长感 |
| 文化探索 | 专题数量有限，缺少历史星图、真实天体影像和星图文化图层 | 中文星空特色尚未充分进入核心交互 |
| 工程质量 | 缺少真实 DOM/E2E、视觉回归、axe、Lighthouse 和跨浏览器测试 | 当前自动校验主要覆盖数据和运行时逻辑 |
| 架构 | 大型数据与模块仍通过 `window` 全局变量连接 | 随功能增长，跨模块契约和维护成本会继续上升 |
| 天文边界 | 星座升落仍以中心点近似，DST 回拨重复时刻没有消歧 | 少数地点和日期可能出现不够清楚的结果 |
| PWA | Service Worker 首次安装一次预缓存全部核心资源 | 弱网下安装可能耗时或整体失败 |

## 三、目标用户与核心任务

### 目标用户

1. **第一次看星的人**：希望快速知道今晚能看到什么，不想先学习复杂参数。
2. **入门观测者**：使用裸眼、双筒或小型望远镜，需要可靠的目标、时间和寻找步骤。
3. **文化与历史读者**：关注中国星官、神话、古代天文制度和跨文明比较。
4. **教师与学生**：需要可投屏、可打印、可分享状态的学习材料。
5. **内容贡献者**：希望通过 GitHub 修订数据、补充来源和提交专题。

### 五条必须做顺的核心流程

1. 进入网站后，在 60 秒内生成今晚建议。
2. 搜索任意已收录目标，并在星图、观测和故事之间切换。
3. 把目标加入今晚计划，在离线环境中继续查看和勾选。
4. 按分步路线找到一颗亮星、一个星座或一个深空目标。
5. 记录观测结果，并在下次访问时看到未完成任务、收藏目标和未来天象。

## 四、优先级定义

| 优先级 | 定义 | 处理原则 |
| --- | --- | --- |
| `P0` | 阻碍核心流程、准确性、隐私、移动端或离线可靠性 | 必须在下一阶段优先完成 |
| `P1` | 明显提高观测价值、复访率和内容深度 | 在 `v1.0` 前完成 |
| `P2` | 有趣且有价值，但依赖更多数据、传感器或后端 | 核心流程稳定后再做 |
| 暂缓 | 审核、成本、隐私或维护负担明显高于当前价值 | `v1.1` 或独立项目再评估 |

## 六、v0.7：高可用入口与质量基线

### 版本目标

先让现有功能更容易找到、更容易完成、更容易测试。这个版本不进行大规模视觉重写，也不立即引入 React。

### P0 产品任务

#### 1. 首页任务化

- 第一视口明确提供“今晚看什么”“查找星座”“探索故事”三个入口。
- 默认地点必须明确显示为示例，不让外地用户误以为网站已经定位。
- 信息架构逐步收敛为“今夜、星图、故事、学习、我的、关于”六个稳定入口。
- 移动端形成稳定的主要入口；避免继续向单页底部堆叠功能。
- 星图、故事和观测结果都提供明确的跨模块动作：
  - 在星图中查看。
  - 查看今晚可见性。
  - 加入观测计划。
  - 阅读相关故事。

#### 2. 观测助手简单/高级模式

- 简单模式只保留地点、时间和器材三个主要选择。
- 经纬度、海拔、时区、Bortle、天气授权进入高级设置。
- 拒绝定位后仍能通过城市选择完整使用。
- 高级设置折叠和展开不能改变已输入值。
- 共享链接恢复后，应自动进入与链接相符的模式和状态。

#### 3. 全站统一搜索

- 搜索范围覆盖：
  - 88 个星座及中文、拉丁名、缩写、含义和别名。
  - 具备标识的重点恒星。
  - Messier、NGC 和其他已收录深空目标。
  - 故事专题、人物、时代和文化标签。
  - 月相、流星雨和主要天象类型。
- 支持中文、拉丁名、常见别名和基础拼音索引。
- 搜索结果按“星座 / 恒星 / 深空 / 专题 / 天象”分组，而不是混成一个长列表。
- 每个结果都可恢复深链接，并提供星图、观测和故事入口。

#### 4. 计划与设置持久化

- 引入带 `schemaVersion` 的本地数据结构。
- 保存用户明确允许保存的地点、器材、Bortle、默认图层和收藏。
- 保存今晚计划的目标、勾选状态和生成时间。
- 提供导出、导入和一键清除本地数据。
- 精确坐标继续默认不保存；天气授权不能自动持久化。

### P0 工程任务

#### 1. 测试工具链

- 建立 `package.json` 与锁文件。
- 引入 Playwright，覆盖 Chromium、Firefox 和 WebKit。
- 引入 axe-core 或等价工具做自动无障碍检查。
- 引入 Lighthouse CI 和 gzip 资源预算。
- 继续保留现有零依赖数据校验脚本，避免迁移期间失去已有保护。

#### 2. 六条核心 E2E

1. 搜索、筛选、收藏和星座深链接。
2. 星图键盘移动、缩放和天体资料弹层。
3. 专题搜索、筛选、分页和深链接。
4. 手动地点生成今晚观测结果。
5. 天气成功、失败、超时和关闭授权。
6. 首次离线、重复离线、缓存升级和失败恢复。

#### 3. 数据契约

- 为以下数据建立 JSON Schema 或等价结构校验：
  - `Constellation`
  - `Star`
  - `DeepSkyObject`
  - `StoryTopic`
  - `Citation`
  - `Location`
  - `MeteorShower`
  - `UserPreferences`
  - `ObservationPlan`
- 每条内容增加或统一 `sourceId`、`reviewedAt`、`license`、`confidence` 和说明字段。
- 减少 `constellations.js` 与 `sky-atlas.js` 中的重复字段，派生索引由脚本生成。

#### 4. 首屏与 PWA 优化

- 首页只加载品牌、任务入口和极小摘要。
- D3、全天星图、完整星座档案在接近星图区时加载。
- 深空数据只在开启深空图层或打开相关资料时加载。
- Service Worker 只强制预缓存应用外壳。
- 星图、故事和观测资料改为可选离线包，显示大小、版本和最后更新时间。
- 新 Service Worker 不自动无提示替换正在使用的页面，更新前让用户确认。

### v0.7 验收标准

- 5 名第一次使用者中至少 4 人能在 60 秒内生成今晚建议。
- 统一搜索的 100 条基准查询成功率不低于 95%，本地响应低于 100ms。
- 6 条核心 E2E 在 Chromium、Firefox 和 WebKit 中全部通过。
- axe 自动检查无 `serious` 或 `critical` 问题。
- 320px 宽度和 400% 缩放下无横向溢出、遮挡或不可访问控件。
- 首屏资源不超过 300KB gzip；PWA 强制核心包不超过 350KB gzip。
- Lighthouse 移动端 Performance 不低于 85，Accessibility、SEO、Best Practices 不低于 95。
- CI 在 10 分钟内完成。

## 七、v0.8：户外观测与天文精度

### 版本目标

把“最佳时刻”升级为可以真正带到户外执行的整夜计划，并在测试保护完成后建立更稳固的应用内核。

### 迁移策略

- `v0.8` 将迁移到 **Astro + strict TypeScript** 作为必做项，内部沿用 Vite。
- 不引入 React 作为默认前提；Canvas 星图、D3 和 Astronomy Engine 继续作为原生 TypeScript 岛组件。
- `v0.7` 末执行迁移门槛评审；未满足时应延后 `v0.8`，而不是一边扩充大功能一边无保护重写。
- 如果独立静态页面尚未进入实现，可先用 Astro 单页或纯 Vite + TypeScript 完成模块迁移；`v0.9` 再完成全部静态路由。
- 进入迁移前必须满足：
  - 六条核心 E2E 已稳定。
  - 天文黄金样例已建立。
  - 数据 Schema 已固定。
  - 旧 URL、收藏、地点和本地计划有迁移测试。
  - 首屏和离线资源预算已经明确。

### P0 观测功能

#### 1. 整夜时间轴

- 在一条时间轴上显示：
  - 日落、暮光和天文黑夜。
  - 月亮高度与照明影响。
  - 云量、降水、风速和预报更新时间。
  - 已选目标的高度曲线和最佳窗口。
- 用户拖动时间轴时，地平星图、行星位置和清单同步更新。
- 推荐分数拆解为黑暗、月光、云量、目标高度、器材和光污染，不只显示一个黑盒分数。

#### 2. 目标选择与高度曲线

- 用户可主动选择星座、恒星、行星和深空目标加入计划。
- 每个目标显示最佳时间、方位、高度、剩余窗口、器材、难度和推荐原因。
- 支持自定义最低高度和简单地平遮挡配置。
- 星座可见性从单一中心点升级为“主要亮星可见比例 + 边界采样”，同时保留估计口径说明。

#### 3. 户外夜视模式

- 红黑低亮主题，切换过程中不出现白色闪烁。
- 关键按钮和状态使用大触点、大字号和非颜色提示。
- 支持全屏和 Screen Wake Lock；明确说明耗电并提供关闭入口。
- 提供离线预检：星图、计划、故事包、最后天气预报是否可用。
- 离线天气必须显示缓存时间，不能伪装成实时结果。

#### 4. 天象日历与 ICS

- 提供未来 7、30 和 90 天视图。
- 在现有月相、流星雨和合月基础上，逐步加入：
  - 行星冲与大距。
  - 日月食。
  - 主要掩星。
  - 流星雨辐射点高度与月光影响。
- 支持 `.ics` 导出到系统日历。
- 首版不做 Web Push，避免在静态站点上引入不可靠的后台通知方案。

#### 5. 本地观测记录册

- 从今晚计划一键生成记录。
- 记录目标、看到/未看到、时间、器材、条件和备注。
- 默认只保存城市或模糊坐标。
- 使用 IndexedDB，支持 JSON、CSV 或 Markdown 导出和无损导入。
- 用户可清除全部记录；若以后支持照片，必须默认移除 EXIF。

#### 6. “我的星空”回访入口

- 汇总收藏目标今晚的可见性和下一次较佳窗口。
- 展示下一场重要天象、上次未完成计划、最近观测记录和继续阅读专题。
- 允许管理常用城市、器材、默认图层、离线资料包和本地隐私设置。
- 所有内容从本地数据生成；在没有账号和后端时也能形成完整回访体验。

### P0 时间与精度任务

- 使用 Temporal/polyfill 或可靠时区库处理 DST。
- 对秋季回拨重复时刻提供“第一次 / 第二次”选择。
- 覆盖不存在的当地时刻、半小时时区、日期变更线、极昼和极夜。
- 建立至少 30 组固定地点和日期的天文黄金样例。
- 使用 JPL Horizons 或等价权威结果做离线 fixture 交叉验证。
- 每个 fixture 必须记录并固定以下计算口径：
  - WGS84 观测者纬度、经度和海拔。
  - 输入的 UTC 时刻、IANA 时区和 DST 重复时刻的 fold 选择。
  - 站心或地心坐标；方位高度和升落默认使用站心结果。
  - 几何或视位置、折射模型、标准地平高度，以及天体中心或上下边缘定义。
  - 时间尺度、参考星历、参考查询参数和 Astronomy Engine 版本。
  - 合月事件默认定义为地心视角下两天体中心角距的局部最小值；当地可见性另行计算。
- 下列容差是同一计算口径下的初始目标，`v0.7` 基线报告可根据权威 fixture 调整，但调整必须记录原因：
  - 太阳、月亮和主要行星方位高度误差不超过 0.2°。
  - 升落时刻误差不超过 2 分钟。
  - 合月等事件时刻误差不超过 5 分钟。
- 天气请求失败时，纯本地天文结果必须继续可用。

### v0.8 验收标准

- strict TypeScript 无错误，核心天文、时间和可见性模块单测覆盖不低于 85%。
- 旧深链接、本地收藏、地点和计划迁移测试 100% 通过。
- 首屏不超过 200KB gzip；进入星图后的附加资源不超过 350KB gzip。
- LCP 不超过 2.5 秒、INP 不超过 200ms、CLS 不超过 0.1。
- 星图拖动和缩放的 p95 帧耗时不超过 32ms。
- 天文黄金样例全部在定义容差内。
- 飞行模式下可以打开已保存计划、星图和记录册，并继续勾选与记录。
- 天气、定位、常亮和全屏权限被拒绝时都有完整替代路径。

## 八、v0.9：趣味探索与中文特色

### 版本目标

形成值得反复访问的探索、学习和记录机制，并把中国星官与跨文化内容真正带进星图核心体验。

### P1 趣味功能

#### 1. “今晚三项任务”

- 根据地点、时间、器材、Bortle、天气和用户水平生成三个可完成目标。
- 每项任务说明为什么值得看、如何找到、预计耗时、难度和错过后的下一窗口。
- 完成状态由用户确认，不假装网站能自动识别用户是否看到目标。
- 挑战进度默认只保存在本机，不使用强制连续签到。

#### 2. 星跳导航与识星训练

- 从一颗明显亮星出发，以 3–8 步寻找星座或深空目标。
- 每一步显示方向、角距离、参考形状和适合视场。
- 手机采用单步推进，不把星图和长说明同时塞进狭窄界面。
- 训练模式支持提示、自检和再次尝试，不采用惩罚式计时。

#### 3. 中国星官与跨文化图层

- 首批建立“HIP 恒星 -> 星官 -> 连线 -> 时代 -> 来源 -> 置信度”的结构化数据。
- 同一天区可以切换 IAU 星座、中国星官和其他可靠传统。
- 不把不同时代星官合并成唯一版本。
- 对争议对应、后期补绘和现代边界差异给出说明。
- 图层按需加载，不进入 PWA 最小核心包。

#### 4. 星空时光机

- 第一阶段限制在 1900–2100 年，显示日期、季节、行星和岁差影响。
- 拖动时间时星图、观测条件和相关历史事件同步变化。
- 古代内容先做“历史记录 + 现代回算”的并列展示，不宣称古图和现代模型完全一致。
- 更早年代必须先验证历法转换、岁差、自行和模型适用范围。

#### 5. 真实影像与目视对照

- 为重点目标加入真实天体影像和历史星图素材。
- 明确区分肉眼、双筒、目视望远镜和长曝光摄影的视觉差异。
- 所有素材逐项记录作者、机构、许可证、原始链接和裁剪说明。
- 手机默认加载缩略图，原图按需加载，不进入默认离线包。

#### 6. 分享星空卡片

- 生成适合竖屏和横屏的观测卡片。
- 可包含模糊地点、时间、地平星图、三个任务、月相和相关故事。
- 使用 Canvas `toBlob` 和 Web Share API，低端设备限制导出分辨率。
- 不在图片中暴露精确坐标或未授权的长篇文本。

### P1 静态内容平台

- 静态生成：
  - `/constellations/<id>/`
  - `/stories/<id>/`
  - `/observe/`
- 每页具有独立 title、description、canonical、Open Graph、JSON-LD 和 sitemap 条目。

## 十一、建议的数据模型

- 计算事件和编辑事件分开保存。
- 算法版本、来源年份、适用地区、容差和是否为常年参考。

### `lessons`

- 年龄段、时长、学习目标、演示步骤、活动、答案和关联星图状态。

### 本地用户数据

- 所有结构具有 `schemaVersion`。
- 具有迁移、导入、导出和删除机制。
- 精确坐标、搜索原文、观测备注和照片不得进入匿名统计。

## 十五、暂缓事项

以下功能暂不进入 `v1.0` 主线：

- 账号、跨设备同步和多人协作收藏。
- 评论、点赞、动态流和公开位置地图。
- AI 自动编写神话或无来源的内容扩写。
- 强制连续签到和纯粹为了时长的游戏化。
- 从零开发相机识星或把 AR 设为主入口。
- 实时卫星追踪和与主线无关的 3D 太阳系。
- 依赖后端定时任务的 Web Push 通知。

这些能力需要额外后端、审核、存储、隐私和时效数据维护。只有当核心流程、内容治理和真实用户需求得到验证后，才单独立项。
